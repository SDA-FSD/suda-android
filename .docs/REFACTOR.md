# Roleplay 준비 리팩토링 작업 분해

## 공통 배경
- 본 프로젝트는 Flutter 기반 영어 회화 학습 앱이며, 롤플레이 기능이 핵심 흐름이다.
- 현재 구조는 다음 영역에서 변경 비용이 높다: 토큰 갱신 로직 이원화, `main.dart` 책임 집중, 라우팅 분산, 로컬라이즈 텍스트 추출 중복, Roleplay 상태관리 빈약, UI-API 결합.
- 앞으로 여러 롤플레이 기능이 추가될 예정이므로, 변경 지점을 줄이고 확장 가능성을 높이는 것이 목표다.

## 공통 목표
- 변경 시 수정 지점을 최소화하고 책임을 분리한다.
- 오버-일원화로 인한 단일 파일 부담을 줄인다.
- 롤플레이 기능 확장 시 재사용 가능한 구조를 우선한다.

## 공통 전제/규칙
- 리포의 사실 기준은 `.docs/CONTEXT.md`.
- 변경이 발생하면 관련 문서도 갱신한다.
- 불필요한 리팩토링 금지, 지정된 범위 내에서만 수행.
- 다중 파일 수정 시 사전 확인 절차 준수.

## 작업 분할 (Agent 단위)

### T1. 토큰 갱신 단일화
- 목표: 토큰 갱신 경로를 하나로 통합하여 레이스/중복을 제거.
- 범위: `TokenRefreshService` 또는 `_TokenRefreshManager` 중 하나를 표준으로 정의하고 나머지 경로를 정리.
- 산출물: 단일 갱신 흐름 설계 요약, 변경된 코드, 관련 문서 업데이트.
- 전제: 백그라운드 타이머 갱신 유지 여부 결정 필요.
- 상태: Known-issue 처리 함.

### T2. API 클라이언트 분리(구조화)
- 목표: `suda_api_client.dart`의 책임 분리로 유지보수성 개선.
- 범위: DTO/모델, HTTP 클라이언트, API 메서드 레이어 분리.
- 산출물: 파일 구조 제안, 단계별 분리 PR.
- 전제: 기존 호출부 영향 최소화 원칙 유지.
- 상태: 완료.

### T3. 라우팅 중앙화
- 목표: 화면 전환 로직을 한곳에서 관리.
- 범위: Roleplay 흐름(overview/opening/playing/ending/failed) 중심으로 라우터 정의.
- 산출물: App Router 설계/적용, 기존 화면에서 직접 `MaterialPageRoute` 제거.
- 전제: 네비게이션 정책 합의 필요(이름 기반 vs 타입 기반).
- 상태: 완료.

### T4. Roleplay 상태 관리 계층 도입
- 목표: Roleplay 진행 상태를 일관되게 관리.
- 범위: `RoleplayStateService` 확장 또는 `ChangeNotifier` 기반 컨트롤러 도입.
- 산출물: 상태 구조 정의, 화면 간 상태 공유 방식 정리.
- 전제: 상태 관리 라이브러리 도입 여부 결정(Provider/Riverpod 등).

### T5. 로컬라이즈 텍스트 추출 유틸 통합
- 목표: 언어 선택 로직 중복 제거.
- 범위: `SudaJson` 기반 공통 유틸 함수 설계 및 적용.
- 산출물: 공통 유틸, 기존 중복 코드 제거, 사용 가이드.
- 전제: 현재 언어 우선순위(현재 언어 → en → first) 유지.
- 상태: 완료.

### T6. UI-API 결합 완화
- 목표: 화면에서 직접 API 호출하는 패턴 감소.
- 범위: Roleplay 관련 API 호출을 repository/usecase로 분리.
- 산출물: Roleplay 전용 레이어 설계 및 적용.
- 전제: 에러 처리 정책(공통 에러/토스트/다이얼로그) 합의.

### T7. 앱 부트스트랩/인증 책임 분리
- 목표: `main.dart`의 비대화 해소.
- 범위: 인증/버전체크/동의 플로우를 별도 서비스 또는 컨트롤러로 분리.
- 산출물: 부트스트랩 흐름 다이어그램, 분리된 클래스/서비스.
- 전제: 초기 진입 플로우 변경 최소화.
